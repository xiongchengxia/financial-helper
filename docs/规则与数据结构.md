# 规则与数据结构说明

本文档与 `contracts/` 目录下的 JSON Schema 对齐，供前后端与识别服务共用契约。版本：`1.0.0`（与导出包 `exportVersion` 一致）。

---

## 1. 标识与通用约定

| 字段 | 说明 |
|------|------|
| `id` | UUID v4 字符串 |
| `createdAt` / `updatedAt` | ISO 8601 日期时间，UTC（`Z` 结尾） |
| 金额 | `number`，单位为元，保留最多 2 位小数；运算用「分」整数可避免误差，当前 Schema 仍用元 |
| `confidence` | `0`～`1` 的浮点数，表示识别或分类置信度 |

## 3. 枚举

### 3.1 `RecognitionTaskStatus`

| 值 | 含义 |
|----|------|
| `pending` | 已创建，等待处理 |
| `processing` | 识别/解析中 |
| `completed` | 成功，已生成 Document |
| `failed` | 失败，`errorMessage` 有值 |

### 3.2 `DocumentKind`

| 值 | 含义 |
|----|------|
| `invoice` | 增值税发票类 |
| `bank` | 银行流水/回单类 |
| `other` | 未分类或其它 |

### 3.3 `InvoiceType`（票面类型，业务子类）

| 值 | 说明 |
|----|------|
| `vat_special` | 增值税专用发票 |
| `vat_normal` | 增值税普通发票 |
| `vat_electronic` | 增值税电子发票等 |

### 3.4 `RiskLevel`

| 值 | 建议处理 |
|----|----------|
| `info` | 记录即可 |
| `warn` | 人工复核 |
| `block` | 阻断入账或强提示 |

### 3.5 `BusinessEventType`（驱动入账规则）

| 值 | 适用文档 |
|----|----------|
| `PURCHASE_WITH_VAT` | 采购取得增值税发票（专票进项） |
| `BANK_PAYMENT` | 银行支出（贷记银行存款等） |
| `BANK_RECEIPT` | 银行收入 |

## 4. 合规规则（机器可执行编号）

规则引擎输出：`passed: boolean` + `hits[]`（每条含 `ruleId`、`passed`、`message`、`evidence`）。

**整体合规** = 全部 `RULE_INV_*` 通过 **且** 不存在 `level=block` 的风险（如 `RISK_DUPLICATE`）。形式规则全通过仍可能因阻断风险而 `passed=false`。

### 4.1 发票形式合规（`RULE_INV_*`）

| ruleId | 说明 | 不通过时典型原因 |
|--------|------|------------------|
| `RULE_INV_REQUIRED_FIELDS` | 号码、开票日期、购销双方名称、价税核心金额齐全；**纸质/非数电票**另需发票代码 | 缺字段；`vat_electronic` 豁免 `invoice_code` |
| `RULE_INV_TAX_ID_FORMAT` | 购销方税号长度在 15/18/20 之一（简化校验） | 税号位数异常 |
| `RULE_INV_AMOUNT_CONSISTENCY` | `total_with_tax ≈ amount_without_tax + tax_amount`（默认容差 0.06 元） | 金额逻辑不符 |
| `RULE_INV_ISSUE_DATE` | 开票日期在合理区间（如 2000-01-01～今天+1 天） | 日期异常 |

### 4.2 发票业务/风险规则（与风险码对应）

风险对象结构见 Schema `risk-flag.schema.json`：`code`、`level`、`message`、`evidence`。

| code | 触发条件（简述） |
|------|------------------|
| `RISK_BUYER_MISMATCH` | 购买方名称或税号与「当前主体」配置不一致 |
| `RISK_AMOUNT_INCONSISTENT` | 与 `RULE_INV_AMOUNT_CONSISTENCY` 同类，提升到风险层展示 |
| `RISK_DUPLICATE` | 数电票按 `invoice_number` 去重；其它票种按 `invoice_code + invoice_number` |
| `RISK_DATE_OUT_OF_RANGE` | 超出配置的报销截止日期或早于 `referenceOrderDate` |
| `RISK_VERIFY_FAILED` | 对接税务查验失败或状态异常（占位，未接查验时可为空） |
| `RISK_SENSITIVE_ITEM` | 明细行商品名称命中敏感词表 |
| `RISK_LOW_CONFIDENCE` | 文档整体 `confidence` 低于阈值 |

**当前主体配置**（API 侧环境变量示例）：

- `COMPANY_BUYER_NAME`：本公司作为购买方时的名称  
- `COMPANY_BUYER_TAX_ID`：本公司税号  
- `REIMBURSE_DEADLINE`：可选，ISO 日期，开票日晚于此则 `RISK_DATE_OUT_OF_RANGE`  
- `SENSITIVE_ITEM_KEYWORDS`：可选，逗号分隔关键词

## 5. 入账规则（模板）

### 5.1 规则表逻辑（概念）

每条规则：`eventType` + 可选条件（如 `invoice.invoice_type === vat_special`）+ `lines[]`。

每行分录：`accountCode`、`accountName`、`side`（`debit`|`credit`）、`amountRef`（引用字段路径）。

### 5.2 内置模板（与代码 `voucher-rules.ts` 一致）

**`PURCHASE_WITH_VAT`（专票）**

| 顺序 | accountCode | accountName | side | amountRef |
|------|---------------|-------------|------|-----------|
| 1 | `1401` | 材料采购 | debit | `invoice.amount_without_tax` |
| 2 | `22210101` | 应交税费-应交增值税-进项税额 | debit | `invoice.tax_amount` |
| 3 | `2202` | 应付账款 | credit | `invoice.total_with_tax` |

**`BANK_PAYMENT`（简化：支出记费用）**

| 顺序 | accountCode | accountName | side | amountRef |
|------|---------------|-------------|------|-----------|
| 1 | `6601` | 销售费用 | debit | `bank.payment_amount` |
| 2 | `1002` | 银行存款 | credit | `bank.payment_amount` |

> 科目仅为示例，上线前需替换为企业会计科目表。`amountRef` 路径与 `invoice.schema.json` / `bank-transaction.schema.json` 字段一致。

## 6. 导出包（全量凭证）

## 8. HTTP 接口（与 OpenAPI 对齐）

见 `contracts/openapi.yaml`。MVP 提供：

- `POST /v1/tasks`：multipart 上传文件，可选 `mock_scenario` 模拟识别结果  
- `GET /v1/tasks/{id}`：任务状态与关联 `documentId`  
- `GET /v1/documents`、`GET /v1/documents/{id}`  
- `GET /v1/vouchers`、`GET /v1/vouchers/{id}`  
- `GET /v1/export/vouchers`：全量 JSON 导出  

详细请求响应字段以 OpenAPI 为准。
