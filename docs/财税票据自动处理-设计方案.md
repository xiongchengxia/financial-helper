# 财税票据自动处理 — 设计方案

面向：**手机拍照 / 上传图片** → **识别发票与银行流水** → **发票合规审核与风险标记** → **按入账规则生成会计凭证** → **结构化全量导出**。

---

## 1. 总体架构（建议）

```
┌─────────────┐    ┌──────────────┐    ┌─────────────────┐
│ 客户端上传   │───▶│ 媒体与任务队列 │───▶│ 分类与识别管线    │
│ (H5/小程序/App)│    │ (对象存储+异步) │    │ OCR + 专用解析   │
└─────────────┘    └──────────────┘    └────────┬────────┘
                                                 │
         ┌───────────────────────────────────────┼───────────────────────────┐
         ▼                                       ▼                           ▼
┌─────────────────┐                   ┌─────────────────┐           ┌─────────────────┐
│ 发票合规与风险   │                   │ 流水结构化       │           │ 入账规则引擎     │
│ (规则+可选查验)  │                   │ (对手方/金额/摘要)│           │ (模板→分录)      │
└────────┬────────┘                   └────────┬────────┘           └────────┬────────┘
         │                                       │                           │
         └───────────────────────┬─────────────┴───────────────────────────┘
                                 ▼
                    ┌─────────────────────────┐
                    │ 凭证存储 + 版本 + 导出   │
                    │ (JSON/Excel/对接格式)   │
                    └─────────────────────────┘
```

- **异步**：图片识别耗时高，上传后返回 `task_id`，轮询/WebSocket 拉结果。  
- **可观测**：每步保留原始图、识别 JSON、规则命中记录，便于审计与纠错。

## 2. 能力拆解

### 2.1 上传与预处理

| 项 | 说明 |
|----|------|
| 格式 | JPG/PNG/WebP；单张或多张；可选 PDF 转图 |
| 预处理 | 纠偏、去边、增强对比度、大图缩放（控制 OCR 成本与延迟） |
| 安全 | 病毒扫描、大小限制、敏感字段落库加密（可选 KMS） |

### 2.2 文档分类

在通用 OCR 前先粗分：**增值税发票类** / **银行回单或流水截图** / **其他**。  

可用：轻量分类模型，或启发式（长宽比、关键字「发票代码」「价税合计」「工商银行」等）+ 置信度。

### 2.3 识别层

| 类型 | 推荐做法 |
|------|----------|
| **增值税发票** | 优先使用云厂商「增值税发票」专用识别（字段齐、省规则）；备选开源 OCR + 版式规则 |
| **银行流水/回单** | 各行版式差异大：模板库（银行+版式版本）+ 通用 OCR；或用户框选关键区辅助 |

输出统一为 **结构化 JSON**（见第 4 节），与版式/供应商解耦。

### 2.4 发票合规审核与风险标记

**合规（形式 + 可选实质）**

- 必填字段完整：代码、号码、开票日期、校验码（普票）、购买方名称/税号、销售方、金额、税额、价税合计、税率等（按票种裁剪）。  
- 格式校验：税号 15/18/20 位、日期合理、金额与税额逻辑关系。  
- **可选**：对接国家税务总局「发票查验」类能力（注意频次、验证码、政策变更；建议异步 + 缓存 + 人工兜底）。

**常见风险点（示例，可配置为规则 ID）**

| 风险码 | 说明 |
|--------|------|
| `RISK_BUYER_MISMATCH` | 购买方名称/税号与当前主体配置不一致 |
| `RISK_AMOUNT_INCONSISTENT` | 价税合计 ≠ 金额+税额（容差外） |
| `RISK_DUPLICATE` | 与库中已报销发票代码+号码重复 |
| `RISK_DATE_OUT_OF_RANGE` | 开票日期早于合同/订单日期或超报销期限 |
| `RISK_VERIFY_FAILED` | 查验不通过或状态异常（红冲、作废等，以查验结果为准） |
| `RISK_SENSITIVE_ITEM` | 品名命中敏感词（礼品、烟、酒等，可自定义） |
| `RISK_LOW_CONFIDENCE` | OCR 关键字段置信度低于阈值 |

每条风险：`code`、`level`（info/warn/block）、`message`、`evidence`（字段名、识别值、规则版本）。

### 2.5 会计凭证自动生成

**原则**：业务事件 → **入账规则（可配置）** → **借贷分录**；不把硬编码分录散落在业务代码里。

建议模型：

- **业务事件类型**：如 `PURCHASE_WITH_VAT`、`BANK_PAYMENT`、`BANK_RECEIPT` 等。  
- **规则**：条件（科目维度、发票类型、费用类型、部门等）+ **分录模板**（多行：科目、借贷、金额表达式）。  
- **金额表达式**：引用结构化字段，如 `invoice.amount_excl_tax`、`invoice.vat_amount`、`bank.amount`。

示例（采购取得专票，简化）：

| 行 | 科目 | 借/贷 | 金额来源 |
|----|------|-------|----------|
| 1 | 库存商品 / 费用 | 借 | 不含税金额 |
| 2 | 应交税费-应交增值税-进项税额 | 借 | 税额 |
| 3 | 应付账款 / 银行存款 | 贷 | 价税合计 |

银行流水可按「摘要/对手方/收支方向」映射到费用或往来科目（需产品侧定义优先级与默认科目）。

**输出**：一张「凭证」含：凭证字、号、日期、附件张数、分录行列表（科目编码、名称、借方、贷方、辅助核算）、来源 `document_ids`。

### 2.6 结构化全量导出

- **推荐主格式**：JSON Lines 或单文件 JSON Schema 固定版本，便于对账与二次开发。  
- **财务常用**：按模板导出 Excel（凭证主表 + 分录明细表）。  
- **扩展**：预留金蝶/用友等导入字段映射表（列名映射配置化）。

导出应支持：**时间范围、状态筛选、含原始识别快照与风险列表**（便于审计）。

## 3. 核心数据流（概念实体）

- `Media`：原图存储路径、hash、上传时间。  
- `RecognitionTask`：状态、类型、错误信息、耗时。  
- `Document`：类型（invoice/bank/other）、结构化字段 JSON、置信度、关联 Media。  
- `ComplianceResult`：发票 ID、通过/不通过、规则命中列表。  
- `RiskFlag[]`：见上。  
- `Voucher` + `VoucherLine[]`：凭证头与分录。  
- `ExportJob`：导出任务与下载链接（大文件异步）。

---

## 4. 发票 / 流水 结构化字段（建议最小集合）

**发票（增值税类，字段名示例）**

- `invoice_code`, `invoice_number`, `issue_date`  
- `buyer_name`, `buyer_tax_id`, `seller_name`, `seller_tax_id`  
- `amount_without_tax`, `tax_amount`, `total_with_tax`  
- `tax_rate`, `invoice_type`（专票/普票/电子等）  
- `items[]`：`name`, `spec`, `unit`, `quantity`, `unit_price`, `amount`, `tax_rate`, `tax_amount`  
- `check_code`（普票后六位场景）  
- `raw_ocr` / `provider_payload`（可选，调试用）

**银行流水/回单**

- `bank_name`, `transaction_time`  
- `counterparty_name`, `counterparty_account`（若有）  
- `debit_amount`, `credit_amount`, `balance`（按实际版式）  
- `summary`, `serial_no`  
- `currency`

## 5. 技术选型参考（按团队熟悉度选）

| 层级 | 选项 |
|------|------|
| 后端 | Node（Nest/Fastify）、Python（FastAPI）、Java（Spring）均可；重规则与批处理时 Python 迭代快 |
| 任务队列 | Redis + RQ/Celery、或云厂商消息队列 |
| 存储 | PostgreSQL + 对象存储（S3/OSS/COS） |
| OCR/发票识别 | 阿里云/腾讯云/华为云「增值税发票识别」；或 PaddleOCR + 自维护版式 |
| 前端 | React/Vue H5 + 相机调用；或 Flutter/React Native |

---

## 6. 实施阶段建议

1. **MVP**：上传 → 异步识别 → 发票/流水 JSON 落库 → 简单列表展示。  
2. **合规与风险**：规则引擎 v1 + 风险列表 UI；查验接口可开关。  
3. **凭证**：科目表 + 少量硬编码模板验证借贷平衡，再规则配置化。  
4. **导出**：JSON + Excel；权限与审计日志。

---

## 7. 合规与免责提示

- 发票查验、涉税处理以**主管税务机关要求**为准；系统输出为辅助，**入账与申报责任**仍由企业/会计人员承担。  
- 存储与传输需满足个人信息与财务数据安全规范（访问控制、脱敏、留存周期）。
